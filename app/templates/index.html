Staking / unstaking gate (contract)

Not a secure context — you are on plain http:// to a non-localhost host. Browsers set isSecureContext === false, and wallet extensions (Talisman, MetaMask, …) usually do not inject an EVM provider on this origin. That is why Wallet debug shows no talismanEth / ethereum / EIP-6963.
Fix (pick one):
  • HTTPS — Put the UI behind TLS (e.g. nginx/Caddy + Let’s Encrypt, or a cert you trust).
  • SSH tunnel → localhost — From your PC: ssh -L 19000:127.0.0.1:19000 user@212.56.40.85 then open http://127.0.0.1:19000/ui (loopback counts as a secure context for wallets).
  • CLI — Use cast send / Foundry or Python with your owner key to call setStakingGatePasswordHash / setStakingUnstakingEnabled (no browser wallet).
On your phone — Mobile Safari/Chrome use the same rule: http://<server-ip>:port is not a secure context, so in-page wallets usually will not work.
  • Best: Put the UI on https:// with a domain (Let’s Encrypt). Open that URL on the phone — same as desktop.
  • Wallet in-app browser: In MetaMask Mobile (or another EVM wallet app), open the Browser tab and go to your HTTPS UI URL; the app injects the provider there. Plain HTTP to an IP in the wallet browser still fails the same way.
  • Talisman on phone: If you only have Talisman as a browser extension (desktop), use a mobile EVM wallet’s in-app browser for gate txs, or run gate setup once from a PC with tunnel/HTTPS.
  • Advanced: Android Termux can run ssh -L 19000:127.0.0.1:19000 user@server then open http://127.0.0.1:19000/ui on the phone — loopback is a secure context.
Gate password is never sent to this app’s server. Use an Ethereum (ETH) account in Talisman (Add account → Ethereum); Substrate-only accounts will not expose an EVM provider. We detect window.talismanEth, window.ethereum, multi-wallet ethereum.providers, and EIP-6963. Match the chain to this deployment, then sign in the extension.
